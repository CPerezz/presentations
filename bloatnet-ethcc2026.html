Contract Storage Size

$ benchmark --category storage-scaling 7 / 26
$ sload --storage-size 4MB..50GB 8 / 26
$ sstore --storage-size 4MB..50GB 9 / 26
// STORAGE SIZE

9–64x Degradation

  • "Cache cliff" between 4.23 MB and 1 GB — cache hit rates drop from ~81% to ~63%
  • Reth SLOAD: 63.8x degradation (MDBX page faults when data exceeds page cache)
  • Nethermind most resilient: 9.2x SLOAD, 6.9x SSTORE — stays under 2s at ALL sizes
  • Current gas is flat (2,100 per SLOAD) despite 9–64x cost variation
[CRITICAL]

Gas pricing must reflect contract storage size AND depth. EIP-8032 addresses size. Depth is harder — trie morphology only known after DB commit. Non-trivial to price correctly.

// STATE SIZE

2x State ≠ 2x Slower

  • Read dominance: State reads consume 50–85% of block time. Read cost is logarithmic in state size.
  • Logarithmic MPT cost: Doubling state adds ~1 trie level ≈ negligible per-operation impact.
  • Variance caveat: N=1 runs per benchmark. Ratios show variance, not definitive trends.
  • Nethermind exception: 1.38x slower, 80% benchmarks degraded — early architectural signal?

284 GB→600 GB is safe. 600 GB→3 TB could hit hardware cliffs. We need to keep measuring.

More Reads = Worse Throughput

  • At 90% SLOAD: Geth drops from 240 to 84 Mgas/s
  • Real-world DeFi is read-heavy — sits in the worst-performing region
  • Write-only appears faster — but writes are more aggressively priced (SSTORE: 20K+ gas vs SLOAD: 2,100). Fewer ops fit per block = less I/O = faster processing. If priced equally, reads would win.

// CONCLUSIONS

Key Takeaways

  • State size growth is not yet a performance concern. 2x state showed negligible impact. More worried about hardware limits (disk fullness, page faults at 3+ TB) than execution performance.
  • Contract storage size is a major concern. 9–64x degradation and current pricing doesn’t reflect it. Gas is flat at 2,100 regardless of contract storage size.
  • Reads dominate performance bottlenecks. State reads are 50–85% of block time. But reads can’t be upriced — they’re the core operation that makes Ethereum work.
  • Bytecode size increases need careful evaluation. Some clients show page-size dependent read speeds. We can’t blindly move to 256 KB without benchmarking all clients.
  • Deep reads/writes are dangerous and unpriced. Trie-depth attacks cause multi-second blocks. No pricing method accounts for depth — trie morphology is only known after committing to DB.
