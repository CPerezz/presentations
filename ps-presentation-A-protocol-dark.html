Today

The Status Quo

  • Every validator holds ~640 GB of state
  • Full re-execution of every transaction
  • State replicated across ~10,000+ validators worldwide
  • Geographic distribution makes censorship nearly impossible
10,000+

copies of Ethereum's state, distributed globally

02 / 18
The Shift

ZKEVMs Change Everything

  • A SNARK proof replaces re-execution
  • Validators verify blocks in milliseconds — without holding ANY state
  • The rational choice? Drop state entirely.

“If verification is free,
why pay for state?”

The Centralization Trap

  • State concentrates in ~10–50 operators (builders + RPC providers)
  • 10,000 → 50 — a 99.5% reduction in state replication
  • Infura, Alchemy, and a handful of builders become the state layer

10,000+ validators with state

Post-ZKEVM

04 / 18
The Crisis

Invisible Censorship

FOCIL (inclusion lists) needs transactions. Transactions need state.

Government pressures a provider to stop serving state for targeted addresses → users can't build transactions → FOCIL includers never see those transactions → censorship achieved without touching the protocol

FOCIL doesn't matter if nobody serves you state to build transactions.
05 / 18
Dead End

State Expiry Is Not the Answer

  1. UX nightmare: your account "expires" after 6 months
  2. Protocol complexity explodes — address space extension, resurrection mechanics
  3. Doesn't solve centralization — wallets still need expired state from the same providers
“State expiry adds protocol complexity. The state remains centralized regardless.”

Architecture

How It Works

  1. Snap sync: download full account trie + selected contract storage
  2. Gap fill: process BALs from pivot to HEAD
  3. Follow tip: apply BAL diffs per block — no re-execution
  4. Serve proofs: generate MPT proofs for tracked state
~59 GB

account trie + USDC + DAI vs ~640 GB full node — 83% reduction

Censorship Resistance

Why PS Matters for FOCIL

  • EIP-7805: 16 includers per slot select transactions from mempool
  • Includers need account data to pick valid transactions
  • Valid txs in ILs = harder for builders to justify exclusion
  • PS drops hardware from ~640 GB to ~50–80 GB → dramatically expands the includer pool
Account data makes includers effective, not just eligible.

Economics

State Markets

  • Emergent, out-of-protocol — many flavours
  • Load balancer / marketplace: PS nodes register tracked contracts, queries get routed
  • Hot state = cheap (many nodes have it). Cold state = premium (few holders).
  • The attestation analogy: one query ≈ nothing. Millions/year ≈ real revenue.

VOPS

The Validator Floor

Validity-Only Partial Statefulness: the minimum viable state for a useful validator.

  • Just account quadruplets (balance, nonce, codeHash, storageRoot)
  • Enables: mempool management + FOCIL inclusion + SNARK verification
  • Any PS node is also a VOPS node — VOPS is a subset
  • Post-ZKEVM, this is the minimum to be a useful validator
~8.4 GB

That's all a validator needs to keep Ethereum censorship-resistant.

The Bridge

Witness-Carrying Transactions

  • Frame Txs need state for validation. Solution: attach witnesses.
  • ~12–20 KB per smart wallet tx (Merkle/Verkle proofs)
  • PS node verifies witness against state root, executes VERIFY locally
  • Trade-off: larger txs, but preserves mempool decentralization
“Ethereum doesn't need to choose between AA and statelessness — but they must be co-designed.”
